Тестовое задание для стажировки (системный аналитик)

Задание 1. Анализ требований

Пункт 1. Выявленные проблемы в ТЗ

1. Противоречия

1.1. Пункты 7 и 13
Пункт 7: цена фиксируется при добавлении в корзину.
Пункт 13: при изменении цены в каталоге она автоматически обновляется в корзине у всех пользователей.
Проблема: требования прямо противоположны. Система не может одновременно фиксировать и обновлять цену.

1.2. Отсутствие пункта 12
Проблема: технический недочёт.

2. Неоднозначности

2.1. Пункт 11 (время показа рекламы)
«Реклама должна быть каждый будний день по утрам и вечерам».
Проблема: нет конкретного временного интервала.

2.2. Пункты 10 и 11 (реклама в корзине)
Пункт 10: в корзине может быть реклама других продуктов.
Пункт 11: реклама показывается по утрам и вечерам в будни.
Проблема: реклама это не товар, ТЗ не определяет, где она отображается и что значит «реклама других продуктов».

3. Избыточность

3.1. Пункты 3 и 5
Пункт 3: в корзине не более 5 различных товаров.
Пункт 5: товары в корзине могут быть разные.
Проблема: пункт 5 полностью покрывается пунктом 3 и является лишним.

4. Неполнота

4.1. Пункт 6
«При попытке добавить товар, превышающий лимиты, показывается сообщение "Лимит корзины превышен"».
Проблема: сообщение не конкретизирует, какой именно лимит нарушен.

4.2. Логическая странность (пункты 1 и 4)
Правило «суммарно не более 20 товаров, но при этом можно добавить один товар до 10 штук» является странным. Не описано, зачем это нужно.

Пункт 2. Исправленная версия ТЗ

1. Добавление и изменение товаров

1.1. Пользователь может добавить от 1 до 10 единиц одного товара за одну операцию.

1.2. Пользователь может изменить количество каждого товара в корзине. Минимальное количество — 1. Максимальное — с учётом ограничений из п. 1.3 и 1.4.

1.3. В корзине может находиться не более 5 различных товаров одновременно.

1.4. Суммарное количество всех товаров в корзине не может превышать 20 штук.

1.5. Если при добавлении или изменении количества товара нарушается хотя бы одно из ограничений (п. 1.3 или 1.4), система:
   - не выполняет операцию;
   - показывает сообщение с указанием конкретного лимита: «Нельзя добавить более 5 разных товаров» или «Общее количество товаров в корзине не может превышать 20 штук».

2. Удаление товаров

2.1. Для удаления товара из корзины используется кнопка «Удалить» рядом с каждым товаром.

2.2. При уменьшении количества товара до 0 товар автоматически удаляется из корзины (поведение идентично кнопке «Удалить»).

3. Цены

3.1. Цена товара фиксируется в момент добавления в корзину и не меняется автоматически.

3.2. Если цена в каталоге изменилась, пользователь видит баннер: «Цена на товар X изменилась. Актуальная цена — Y ₽. Обновите товар в корзине».

3.3. Обновление цены — только по действию пользователя.

4. Отображение корзины

4.1. На странице корзины отображается: список товаров, количество, цена за единицу, общая стоимость позиции, итоговая сумма.

5. Реклама (вынесена в отдельный функционал)

5.1. Требования к рекламному блоку — в отдельном ТЗ.

Пункт 3. Вопросы к продукт-менеджеру (в порядке приоритета)

1. Приоритет лимитов: если добавление товара нарушает суммарный лимит в 20 штук, но не нарушает лимит в 5 разных товаров — отказать, предложить уменьшить количество или скорректировать автоматически?

2. Что считается «разными товарами»? Один товар в разном цвете / размере — это один товар или разные?

3. Что из проекта сейчас должно стоять на первом месте? Какие функции критичны для MVP?

4. Должна ли корзина сохраняться после выхода из аккаунта? Если да, то как долго?

5. Что с ценой при скидке? Автоматически применять или только предупреждать?

6. Какие ещё бизнес-правила должны быть? (минимальная сумма заказа, скидки, возрастные ограничения)

7. Где должна находиться реклама? Только в корзине или на других экранах?

8. Существует ли программа лояльности? Влияет ли на цену в корзине?

Задание 2. Проектирование API

1. Пример REST API запроса

GET https://api.petrushka.ru/v1/partners/stores?lat=55.75&lon=37.62

Параметры запроса:
- lat (необязательный) — широта геолокации
- lon (необязательный) — долгота геолокации

Если lat/lon не переданы, сервер определяет город по IP или использует город по умолчанию (Москва).

2. Пример ответа в формате JSON

3. Пояснения

- При клике на плашку магазина открывается external_url с UTM-метками.
- Типы доставки: interval (конкретное время), range (диапазон минут).
- Фронтенд форматирует: interval → «сегодня 21:00-23:00», range → «Быстрая доставка от 20 до 60 минут».
- В случае ошибки: HTTP 4xx/5xx + JSON { "error": "сообщение" }.
- При большом количестве магазинов — пагинация (limit, offset).

Краткое описание:

1. Ручная рассылка: Админ / CRM → Бэкенд → Notification Service → PUSH-провайдер → Мобильное приложение
2. Автоматическая рассылка: Пользователь → Фронтенд → Бэкенд → Notification Service → PUSH-провайдер → Мобильное приложение
3. Notification Service формирует payload, выбирает провайдера по OS (FCM для Android, APNS для iOS)
4. Провайдер доставляет PUSH, возвращает статус
5. Результат логируется на бэкенде

Мои предложения по улучшению (проактивный анализ)

1. По корзине:
   - Текущие лимиты (5 разных товаров, 20 штук) — гипотеза: они могут отсекать часть пользователей с крупными заказами. Стоит проанализировать, как часто пользователи упираются в лимиты и теряет ли на этом выручка.
   - Добавить метрику «средний чек корзины» и «коэффициент конверсии в заказ» в зависимости от близости к лимитам.

2. По API магазинов:
   - Добавить аналитику переходов по external_url: сколько кликов, сколько из них привело к заказам на сайтах партнёров (через UTM и колбэки).
   - Добавить A/B-тестирование порядка магазинов (сортировка по времени доставки, по близости, по рейтингу), чтобы понять, что лучше влияет на клики.

3. По PUSH-уведомлениям:
   - Добавить метрики: доставляемость, открываемость, конверсия в заказ после PUSH.
   - Тестировать разные тексты и время отправки (утро/вечер), чтобы найти оптимальные.

4. По развитию платформы:
   - В перспективе — интернет-магазин (веб-версия), чтобы покрыть десктопных пользователей. Для этого потребуется отдельное ТЗ, но архитектуру (API, PUSH через веб-пуши) можно сделать общей.
